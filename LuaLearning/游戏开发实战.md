## 第一篇  热更新 ——★★★ 游戏核心

### 1.1 为什么游戏需要热更新 ★★★

游戏（尤其线上游戏/手游）运营期要修 bug、改数值、加活动，但不能让玩家退出重启客户端，也不能停服。Lua 作为脚本层天然支持**运行时重新加载代码**——这是它在游戏开发中存在的核心理由之一。

- **客户端**：修 UI/逻辑 bug、加活动玩法，不用重新发包过审
- **服务器**：修逻辑 bug、调数值，不停机

热更新的本质：**替换函数定义，但尽量保留运行时数据状态**。

### 1.2 require 的缓存机制与清空 ★★★

`require` 加载的模块缓存在 `package.loaded` 中，重复 `require` 不会重新执行文件。热更新第一步就是**清缓存**：

```lua
-- 清除指定模块缓存，下次 require 会重新执行
package.loaded["game.skill.fireball"] = nil
require("game.skill.fireball")    -- 重新加载

-- 批量清除（按前缀）
for k in pairs(package.loaded) do
  if string.find(k, "^game%.") then
    package.loaded[k] = nil
  end
end
```

> ★ **对比**：`dofile` 不走缓存，每次都重新执行，适合调试期；生产热更用 `require` + 清缓存更可控（能拿到返回的模块表，便于做就地更新）。

### 1.3 热重载的两种粒度 ★★★

**模块级热更**（整文件重载）：

```lua
function reload(moduleName)
  local old = package.loaded[moduleName]
  package.loaded[moduleName] = nil
  local ok, new = pcall(require, moduleName)
  if not ok then
    package.loaded[moduleName] = old    -- 失败回滚，避免半坏状态
    error(new)
  end
  return new
end
```

**函数级热更 / 就地更新**（保留模块表引用）：

外部持有的往往是 `M.foo` 这种引用，直接替换整个模块表后旧引用仍指向旧函数。更稳妥的做法是**就地更新旧表**：

```lua
function hotfix(moduleName)
  local old = package.loaded[moduleName]
  package.loaded[moduleName] = nil
  local new = require(moduleName)
  for k, v in pairs(new) do
    old[k] = v     -- 把新函数写到旧表，外部持有的旧引用自动生效
  end
  package.loaded[moduleName] = old
  return old
end
```

### 1.4 状态保持与 upvalue 难题 ★★

热更最大难点：**闭包的 upvalue 不会跟着重载**。

```lua
-- 旧版本
local count = 0
function M.tick()
  count = count + 1
  return count
end
```

重载后 `M.tick` 是新函数，但它的 `count` 从 0 重新开始——旧 `count` 的值丢失了。三种应对：

1. **状态外置到模块表/状态表**：不依赖 upvalue 持久化
2. **用 `debug.getupvalue` / `debug.setupvalue` 迁移 upvalue**：把旧函数的 upvalue 拷到新函数
3. **状态与代码分离**：数据放专门的状态表，逻辑只读写它

```lua
-- 用 debug 库迁移 upvalue（简化版）
local function migrateUpvalue(oldFn, newFn)
  local i = 1
  while true do
    local name, value = debug.getupvalue(oldFn, i)
    if not name then break end
    debug.setupvalue(newFn, i, value)
    i = i + 1
  end
end
```

> ★★★ **核心结论**：热更友好代码 = **逻辑函数尽量无状态**，状态集中管理；闭包别持有需要跨热更保留的数据。

## 第二篇  事件系统 ——★★★ 通信核心

### 2.1 观察者模式 ★★★

游戏中对象间极少直接互调，而是通过**事件总线**解耦：发送方 `emit`，接收方 `on` 订阅。例如"玩家死亡"事件，UI、成就、任务系统各自监听，互不依赖。

### 2.2 事件分发器实现 ★★★

```lua
local M = {}
local handlers = {}    -- eventName -> { [handler] = priority }

function M.on(name, handler, priority)
  priority = priority or 0
  local list = handlers[name]
  if not list then list = {}; handlers[name] = list end
  list[handler] = priority
  return handler       -- 返回便于 off
end

function M.off(name, handler)
  local list = handlers[name]
  if list then list[handler] = nil end
end

function M.emit(name, ...)
  local list = handlers[name]
  if not list then return end
  -- 按优先级排序后调用
  local ordered = {}
  for h, p in pairs(list) do ordered[#ordered+1] = {h=h, p=p} end
  table.sort(ordered, function(a,b) return a.p > b.p end)
  for _, item in ipairs(ordered) do item.h(...) end
end

return M
```

使用：

```lua
local Event = require("event")

local token = Event.on("player.die", function(player)
  print(player.name .. " died")
end)

Event.emit("player.die", {name="hero"})
Event.off("player.die", token)
```

> ★ **坑**：在回调里 `off` 自己、或在一个事件中触发另一事件修改同一监听表，会导致遍历错乱。生产实现常用"延迟删除"标记，或在 `emit` 前拷一份监听表再遍历。

### 2.3 C 层事件绑定（引擎/服务器场景）★★

游戏常在 **C 层触发事件、Lua 层处理逻辑**。典型流程：C 定义事件（网络包到达、定时器到期）→ Lua 注册 handler 到 C → C 触发时回调 Lua。

C 侧必须**保存 Lua 函数引用**，不能让它被 GC：

```c
// 注册时把 Lua 函数存到 registry，拿到整数引用
int ref = luaL_ref(L, LUA_REGISTRYINDEX);

// 触发时取回并调用
lua_rawgeti(L, LUA_REGISTRYINDEX, ref);
lua_pcall(L, 0, 0, 0);

// 注销
luaL_unref(L, LUA_REGISTRYINDEX, ref);
```

Lua 侧注册接口：

```lua
-- 引擎/服务器提供的 C 函数
ccb.register("net.login", function(data)
  -- 处理登录
end)
```

> ★★★ **关键**：Lua 注册给 C 的回调，**C 必须用 `luaL_ref` 持引用**，否则 Lua 侧无人引用就会被回收，C 触发时取到空对象崩溃。

## 第三篇  LuaJIT 与 FFI ——★ 进阶性能

### 3.1 LuaJIT vs 标准 Lua ★★

[LuaJIT](http://luajit.org) 是 Lua 5.1 的高性能实现，JIT 编译后可逼近 C 速度。很多游戏用它（Pillars of Eternity、World of Warcraft UI 等）。

| 对比项 | 标准 Lua 5.1 | LuaJIT |
|-------|-------------|--------|
| 语法 | Lua 5.1 | Lua 5.1 + 部分 5.2 扩展 |
| 性能 | 解释执行 | JIT 编译，快 10~100 倍 |
| FFI | 无 | 有，可直接调 C |
| 位运算 | 无 | 有 `bit` 模块 |
| 64bit | 完整支持 | 早期需 `LUAJIT_GC64` |

> ★ **注意**：LuaJIT 锁在 Lua 5.1 语法（加少量 5.2 扩展），不要按 5.3+ 写。游戏选型要确认目标平台允许 JIT——iOS 早期禁 JIT，需 `jit.off()` 退回解释模式。

### 3.2 FFI 基础 ★★

FFI（Foreign Function Interface）让 Lua 直接调 C 函数、操作 C 结构体，**无需写 binding 代码**：

```lua
local ffi = require("ffi")

-- 声明 C 函数
ffi.cdef[[
  int printf(const char *fmt, ...);
  void *malloc(size_t);
  void free(void *);
]]

ffi.C.printf("hello %d\n", 42)

-- 直接操作 C 内存
local p = ffi.C.malloc(100)
ffi.C.free(p)
```

游戏中常用 FFI 调引擎 C 接口、做高性能数值计算（向量/矩阵）。但要警惕：FFI 绕过 Lua 安全层，**内存自己管**，用错易崩溃。

### 3.3 JIT 优化要点 ★

JIT 对代码有偏好，写法影响是否被编译优化：

- **热点循环**会被 JIT，零散调用不会
- **类型稳定**：循环里变量类型别频繁变
- **避免** `debug.sethook`、`string.dump` 等会关闭 JIT 的调用
- 用 `local` 缓存全局函数（`local sin = math.sin`）

## 第四篇  Lua 服务器开发 ——★★ 服务端实战

### 4.1 lua_State 隔离模型 ★★★

游戏服务器（典型如云风的 [Skynet](https://github.com/cloudwu/skynet)）采用**每个服务一个独立 lua_State** 的架构：

- 每个服务（actor）= 一个 lua_State + 一个 C 服务
- 服务间**不共享全局变量**，完全隔离
- 通信靠**消息传递**（send/call），类似 Erlang 的 actor 模型

好处：一个服务崩溃不拖垮全服；状态隔离天然并发安全；可独立热更。

```c
// Skynet 中每个服务创建独立 lua_State
lua_State *L = luaL_newstate();
luaL_openlibs(L);
// 注册该服务专属的 C API（收发消息等）
lua_pushcfunction(L, l_send);
lua_setglobal(L, "skynet_send");
// 各服务互不可见
```

> ★★★ **核心结论**：服务器里不要想着"共享一块大 Lua 表给所有服务用"——违背隔离模型，引入竞态和耦合。跨服务一律走消息。

### 4.2 C 绑定与事件注册 ★★★

服务器分工：**C 负责底层（网络/定时器/内存），Lua 负责业务逻辑**。两种绑定方向：

**1. Lua 主动调 C**（注册 C 函数给 Lua）：

```c
static int l_send(lua_State *L) {
  const char *addr = luaL_checkstring(L, 1);
  size_t len; const char *msg = luaL_checklstring(L, 2, &len);
  skynet_send(...);          // 调底层发送
  return 0;
}

lua_pushcfunction(L, l_send);
lua_setglobal(L, "skynet_send");
```

**2. C 主动调 Lua**（C 触发 Lua 事件）：

服务器收到网络包，分发给对应 Lua handler。**C 侧只暴露一个通用分发入口，Lua 侧维护 handler 表做二级分发**——C 不用关心具体业务类型，新增协议只改 Lua：

```c
void on_net_msg(const char *type, const char *data) {
  lua_State *L = service->L;
  lua_getglobal(L, "_dispatch");     // Lua 侧总分发函数
  lua_pushstring(L, type);
  lua_pushstring(L, data);
  if (lua_pcall(L, 2, 0, 0) != 0)
    fprintf(stderr, "dispatch err: %s", lua_tostring(L, -1));
}
```

Lua 侧：

```lua
local handlers = {}

function _dispatch(type, data)
  local h = handlers[type]
  if h then h(data) end
end

-- 业务注册（新增协议只改这里）
handlers["login"] = function(data)
  -- 处理登录
end
```

### 4.3 协程驱动异步 IO ★★★

服务器大量异步操作（网络 RPC、数据库查询）。用协程把"回调式"变成"同步等待式"，逻辑清晰：

```lua
-- 没有 coroutine：回调地狱
db.query(sql, function(rows)
  cache.set(key, rows, function()
    net.send(addr, result, function()
      -- 越嵌越深
    end)
  end)
end)

-- 有 coroutine：写成同步样
function handle(req)
  local rows  = yield_db("query", sql)    -- 挂起，结果回来再 resume
  yield_cache("set", key, rows)
  yield_net("send", addr, result)
end
```

实现要点：发起异步请求时 `coroutine.yield`，C 层结果回来后 `coroutine.resume` 唤醒。Skynet 的 `skynet.call` / `skynet.send` 就是这套机制。

调度器需维护 **coroutine 池**（一个请求一个协程，处理完丢弃）与 **coroutine ↔ 请求 id 的映射**，以便回调时找到对应协程唤醒。

> ★★★ **核心结论**：游戏服务器 Lua 的精髓 = **actor 隔离（多 lua_State）+ 协程化异步（同步写法）+ 热更新（不停服改逻辑）**。

### 4.4 服务器热更新 ★★

服务器热更比客户端要求更高（不能影响在线玩家）。常见做法：

1. **模块级热更**：同客户端，清 `package.loaded` 重载，但要先处理 upvalue 迁移
2. **服务级热更**：Skynet 支持 `skynet.inject` 往运行中的服务注入一段 Lua 代码执行
3. **灰度**：新逻辑先在一个服务实例跑，验证后全量

热更时要特别注意**正在执行的协程**：重载时若有协程挂在 `yield`，旧协程仍引用旧函数在执行，需等其自然结束；新请求走新逻辑。

## 附录：游戏开发速查

### 游戏开发常用模式

| 模式 | 用途 | 关键点 |
|------|------|--------|
| 热更新 | 运行时改逻辑 | 清 `package.loaded`、状态外置 |
| 事件总线 | 对象解耦通信 | `on`/`emit`/`off`、回调中改监听表要小心 |
| 协程异步 | 技能/对话/IO 流程 | `yield` 等结果、调度器 resume |
| Actor 隔离 | 服务器并发 | 每服务一 `lua_State`、消息通信 |
| C 绑定 | 性能/底层能力 | `luaL_ref` 持引用、registry 存状态 |

### 游戏开发坑合集

| 坑 | 说明 |
|----|------|
| 热更后状态丢失 | upvalue 不迁移，状态要外置 |
| Lua 回调被 GC | C 侧必须 `luaL_ref` 持引用 |
| 跨服务共享全局表 | 违背隔离，走消息 |
| 回调中 `off` 自己 | 遍历错乱，用延迟删除 |
| 每帧创建大量 table | GC 卡顿，用对象池 |
